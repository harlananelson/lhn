## 1. System Overview
`lhn` is a Python/PySpark abstraction layer designed for high-throughput healthcare cohort analysis. It uses a **Metadata-Driven Factory Pattern** to dynamically instantiate database connections based on YAML configurations.

**Core Philosophy:**
* **Physical Data** is defined in `000-config.yaml` (The "What").
* **Processing Logic** is defined in `config-global.yaml` (The "How").
* **Runtime Objects** (`e`, `r`, `db`) are generated by merging these two files.

## 3\. Configuration & Schema Resolution (The "Magic")

The system connects specific database names to Python variables through a two-step lookup process.

### Step A: Define Physical Schemas (`000-config.yaml`)

You define **where** the data lives.

```yaml
schemas:
  # The physical database name in Hive/Spark
  projectSchema: "sicklecell_rerun"
  RWDSchema: "iuhealth_ed_data_cohort_202306"
```

### Step B: Define Processing Rules (`config-global.yaml`)

The system defines **how** to handle that schema and **what** to name the Python object.

```yaml
callFunProcessDataTables:
  projectcallFunc:
    schema_type: 'projectSchema'  # <--- Looks up this key in 000-config.yaml
    property_name: 'db'           # <--- Creates this Python object (resource.db)
    type_key: 'proj'              # <--- Internal reference
    data_type: 'projectTables'    # <--- Which table list to load
```

### Step C: The Result (Python Object Model)

When `Resources` initializes:

1.  It reads `projectcallFunc`.
2.  It sees `schema_type: 'projectSchema'`.
3.  It looks up `projectSchema` in `000-config.yaml` -> gets `"sicklecell_rerun"`.
4.  It creates a `DB` object connected to `"sicklecell_rerun"`.
5.  It assigns that object to `self.db` (based on `property_name: 'db'`).

**Resulting Objects:**

| Python Variable | Source Config Key | Physical DB (Example) | Purpose |
| :--- | :--- | :--- | :--- |
| **`db`** | `projectSchema` | `sicklecell_rerun` | Project-specific tables (Extracts). |
| **`r`** | `RWDSchema` | `iuhealth...202306` | Read-only RWD tables (Clinical Event, Encounter). |
| **`iuh`** | `IUHSchema` | `iuhealth_prime` | IUHealth operational data. |
| **`e`** | `projectTables` | N/A | **Extract Controller**: Holds definitions (codes) defined in YAML. |

## 4\. The Analysis Workflow

### 1\. Define Cohort Logic (`000-config.yaml`)

Define your code sets in the `projectTables` section.

```yaml
projectTables:
  sicklecodes:
    label: "Codes to search for Sickle Cell"
    dictionary:
      scd: "D57.1|D57.2"
    listIndex: "codes"
    sourceField: "conditioncode_standard_id"
```

### 2\. Execute in Notebook

Use the injected objects (`e`, `r`) to process data.

```python
# 1. Load the Dictionary
# Converts the YAML 'dictionary' into a DataFrame (e.sicklecodes.df)
e.sicklecodes.dict2pyspark()

# 2. Extract Entities
# Joins the codes (e.sicklecodes) against the massive RWD condition table (r.conditionSource)
e.sickleConditionEncounter.entityExtract(
    elementList=e.sicklecodes,
    entitySource=r.conditionSource.df
)

# 3. Create Index
# Summarizes to one row per patient (First/Last date)
e.scdpatient.write_index_table(
    inTable=e.sickleConditionEncounter
)
```

## 5\. Advanced Capabilities

  * **Cohort Matching**: `lhn.cohort_matching.call_match_controls` matches cases to controls using Euclidean distance on demographics.
  * **Ontology**: `lhn.discern` maps local codes to standard ontologies.
  * **Features**: `lhn.features` contains `select_only_baseline` to filter clinical data.
