

## P0 (design decisions — surface, do not auto-fix)
- **Pattern 9, "Export to CSV" / "Export to Parquet" (lines under `e.final_cohort.to_csv()`):** The pattern mutates the ExtractItem by assigning `e.final_cohort.df = final_cohort`, then calls `to_csv()` and writes to `e.final_cohort.parquet`. This bypasses the normal create_extract/entityExtract lifecycle that produces an ExtractItem. Decide whether the documented contract is (a) ExtractItems must always be populated via `create_extract`/`entityExtract` (in which case this example is wrong and should be replaced with a freestanding `writeTable` + CSV export), or (b) direct `.df =` assignment is supported and should be documented as a sanctioned escape hatch. Currently the doc contradicts the rest of the guide's "use the method" philosophy.
- **Pattern 5, Demographics Standardization:** The example operates on `r.demoSource.df` directly and reassigns to a local `demo` variable instead of going through an ExtractItem. This contradicts §2 ("use lhn methods … call them on the item `e.X`") and the three-step pattern overview. Decide whether demographics standardization is a sanctioned exception (operating on raw refs) or whether it should be wrapped in an ExtractItem pattern, and state that explicitly.
- **Pattern 7, Time-Window Analysis:** Hand-rolls a join + `F.date_sub`/`F.date_add` filter against `r.conditionSource.df` rather than using an ExtractItem method. If lhn has a window/anchor helper (e.g., something on ExtractItem akin to `entityExtract` with a date window), the example should use it; if not, the doc should explicitly call out that no package helper exists yet so readers don't go hunting. Decision needed.
- **Pattern 9 partitioning advice + Performance Tip #4:** Both recommend `partitionBy='tenant'` unconditionally. For small project outputs this produces tiny-file partitioning that hurts downstream reads. Decide whether to qualify the guidance ("only for large multi-tenant tables") or drop the blanket recommendation.

## P1 (inaccuracies, contradictions, missing critical facts)

Readers who haven't read the linked instructions file will produce malformed archives that pass local extraction but fail on HDL. Fix: inline a minimal 4–6 line skeleton showing the raw cell marker, the triple-quote fence, and a representative YAML key (`title:`, `resources:`) so the requirement is unambiguous.

The actual column-naming rule (label defaults to `index_<code>` / `last_<code>` unless `indexLabel`/`lastLabel` overrides) is not stated. - **Pattern 2 Step 4 comment:** Lists `course_of_therapy, therapy_gaps` as result columns. Confirm these are the actual emitted column names produced by `write_index_table` when `datefieldStop`/`max_gap`/`sort_fields` are set on the ExtractItem — if the actual columns are different (e.g., `n_courses`, `gap_days`), update to match the package output, otherwise the harness column-validation will flag downstream consumers.
- **Pattern 6 Step 4 filter:** Uses `F.col('index_diagnosis') < F.col('index_medication')`, but Step 1/2 created `e.dx_index` and `e.rx_index` with codes that, per the §"output naming" rule, would yield `index_dx` / `index_rx` (or whatever `code:` is set to in config). The example's column names don't follow from its own config. Fix: either make the `code:` values explicit in the example (`code: diagnosis`, `code: medication`) or change the filter to `index_dx` / `index_rx`.
- **Pattern 9 `writeTable` call:** `outTable='project_schema.final_cohort_scd_rwd'` hardcodes a schema string. The doc's own §5 says "no hardcoded absolute paths" and the broader principle is config-driven. Fix: pull the schema from `ctx` (e.g., `f"{ctx.outSchema}.final_cohort_scd_rwd"` or whatever the documented attribute is) and note that the table name suffix should come from the ExtractItem when available.

Clarify whether the freestanding `attrition` import is still part of the public API or whether the method form is canonical; if the latter, drop `attrition` from the import line.

The §"Notebook Authoring Conventions" intro never mentions `Resources` at all, so a reader hitting the legacy note has no context. Fix: either remove the legacy note (the harness will reject `Resources()` anyway) or add a parenthetical: "`Resources()` is rejected by `check_notebook.py`; migrate any inherited notebook to `pipeline_setup` before running the gate."
- **§7 `hdl_run.py` block, `--fix-loop`:** Description says "Add `--fix-loop` with `--validate` to cycle check → 3090 fix → re-check", but the example uses `--all --fix-loop` without `--validate`. Either `--all` implies `--validate` (state that) or the example is wrong (add `--validate`). Fix: add an explicit sentence: "`--all` includes `--validate`, so `--all --fix-loop` is sufficient."
